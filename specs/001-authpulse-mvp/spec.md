# 機能仕様書: AuthPulse MVP — 認証認可イベント収集・集計・異常検知SaaS

**Feature Branch**: `001-authpulse-mvp`
**作成日**: 2026-03-08
**ステータス**: Draft
**概要**: 既存の Rust + Envoy gRPC マイクロサービス基盤を拡張し、
認証認可イベントを収集・集計・監視するSaaSのMVPを構築する。

---

## ユーザーシナリオ & テスト *(必須)*

### User Story 1 — 認証認可イベントの受信と保存 (Priority: P1)

セキュリティ担当者（またはサービス側システム）が、
アプリケーションで発生した認証認可イベント（ALLOW/DENY）を
AuthPulse にリアルタイムで送信し、確実に記録する。

**Why this priority**: イベント収集こそが本サービスの起点であり、
これがなければ集計・異常検知のいずれも成立しない。MVPの核となる機能。

**Independent Test**: gRPCクライアントで `IngestEvent` を呼び出し、
指定したイベントが保存済みとして参照できれば独立して動作確認可能。

**Acceptance Scenarios**:

1. **Given** 有効な `tenant_id` と必須フィールドを含むリクエストを送信したとき、
   **When** `IngestEvent` を呼び出すと、
   **Then** `OK` レスポンスが返り、イベントが永続化される。
2. **Given** `tenant_id` が空のリクエストを送信したとき、
   **When** `IngestEvent` を呼び出すと、
   **Then** バリデーションエラー（`INVALID_ARGUMENT`）が返り、保存されない。
3. **Given** 同一テナントから短時間に大量のリクエストが来たとき、
   **When** レート制限閾値を超えると、
   **Then** `429 Too Many Requests` が返り、既存データは保護される。
4. **Given** 別テナントのデータが存在するとき、
   **When** あるテナントがイベントを送信すると、
   **Then** 他テナントのデータには影響しない（テナント分離）。

---

### User Story 2 — メトリクス集計の参照 (Priority: P2)

セキュリティ運用担当者が、テナント単位・期間指定で
認可成功率・拒否率・レイテンシなどの集計値を取得し、
運用状況を把握する。

**Why this priority**: イベント収集の次に重要なのは「何が起きているか」の可視化。
集計APIがあれば監視ダッシュボードとして機能し始める。

**Independent Test**: テスト用イベントを投入後に `GetMetrics` を呼び出し、
期待する集計値が返れば独立して確認可能。

**Acceptance Scenarios**:

1. **Given** テナントAに複数の認可イベント（ALLOW/DENY混在）が記録されているとき、
   **When** テナントAで期間と集計種別を指定して `GetMetrics` を呼び出すと、
   **Then** 期間内のALLOW数・DENY数・成功率・p50/p95/p99レイテンシが返る。
2. **Given** テナントAのデータが存在するとき、
   **When** テナントBが `GetMetrics` を呼び出すと、
   **Then** テナントBの自分のデータのみが返る（他テナントデータは含まれない）。
3. **Given** 指定期間にイベントが1件もないとき、
   **When** `GetMetrics` を呼び出すと、
   **Then** ゼロ値の集計結果が返る（エラーにはならない）。

---

### User Story 3 — ルールベース異常検知とアラート参照 (Priority: P3)

セキュリティ担当者が、異常な認証認可パターン（短時間での連続拒否、
不審なIPからのアクセス急増など）を検知し、アラートとして確認する。

**Why this priority**: 収集・集計の次のステップとして異常を自動検知することで、
本サービスの監視SaaSとしての差別化価値を生む。

**Independent Test**: 異常検知ルールをトリガーするイベントパターンを送信後に
`ListAlerts` を呼び出し、アラートが生成されていれば確認可能。

**Acceptance Scenarios**:

1. **Given** 5分以内に同一テナントで DENY イベントが閾値（デフォルト: 10件）を超えたとき、
   **When** システムが定期評価を実行すると、
   **Then** 「DENY閾値超過」アラートが生成される。
2. **Given** 同一ユーザーが60秒以内に5回連続で認証失敗したとき、
   **When** システムが評価を実行すると、
   **Then** 「連続認証失敗」アラートが生成される。
3. **Given** アラートが生成された状態で、
   **When** テナントが `ListAlerts` を呼び出すと、
   **Then** 未解決アラートの一覧が返る（他テナントのアラートは含まれない）。
4. **Given** アラートが解決済みとしてマークされたとき、
   **When** `ListAlerts` をデフォルトフィルタで呼び出すと、
   **Then** 解決済みアラートは一覧に含まれない。

---

### User Story 4 — Webhook 通知連携 (Priority: P4)

セキュリティ担当者が、異常検知アラートの発生時に
登録済みの Webhook URL へ即時通知を受け取る。

**Why this priority**: アラートを能動的に確認するだけでなく、
検知後の即時通知により対応速度が向上する。P3完了後に追加する。

**Independent Test**: Webhook受信用エンドポイントを用意し、
アラートトリガー後に Webhook リクエストが届けば確認可能。

**Acceptance Scenarios**:

1. **Given** テナントが有効な Webhook URL を登録しているとき、
   **When** アラートが新規生成されると、
   **Then** 登録 URL に POST リクエストが送信され、アラート内容が含まれる。
2. **Given** Webhook 送信先が応答しない（タイムアウト）とき、
   **When** 送信を試みると、
   **Then** リトライが行われ（最大3回）、失敗は内部ログに記録される。
3. **Given** Webhook URL が未登録のテナントのとき、
   **When** アラートが生成されても、
   **Then** Webhook 送信はスキップされ（エラーにはならない）。

---

### エッジケース

- 同一イベントが重複送信された場合（ネットワーク再試行等）、べき等性を保つか否か？
  → MVP では `trace_id` が同一の場合に重複登録を許容する（後工程で対応）。
- イベントフィールドが欠損している場合は `INVALID_ARGUMENT` を返し保存しない。
- テナントの月間イベント件数がシステム上限を超えた場合の挙動を定義する（MVP は未実装）。
- タイムゾーンが異なるテナントからのタイムスタンプは UTC に正規化して保存する。
- 異常検知ルールの閾値はシステム全体デフォルト値を使用する（テナント別カスタマイズはMVP外）。

## 要件 *(必須)*

### 機能要件

**イベント収集（P1）**

- **FR-001**: システムは `IngestEvent` gRPC エンドポイントを公開し、
  認証認可イベントを受け付けなければならない（MUST）。
- **FR-002**: イベントには以下のフィールドが必須であり、欠損時は拒否しなければならない（MUST）:
  `tenant_id`, `user_id`, `service`, `resource`, `action`, `decision`（ALLOW/DENY）, `timestamp`
- **FR-003**: `reason_code`, `latency_ms`, `source_ip`, `trace_id` はオプションフィールドとして受け付けなければならない（MUST）。
- **FR-004**: システムはテナント単位でレート制限を適用しなければならない（MUST）。
  制限超過時は `429` を返す。
- **FR-005**: テナント間のデータ分離を保証しなければならない（MUST）。

**メトリクス集計（P2）**

- **FR-006**: `GetMetrics` gRPC エンドポイントは `tenant_id`・開始日時・終了日時を受け付け、
  以下の集計値を返さなければならない（MUST）:
  ALLOW数、DENY数、認可成功率、p50/p95/p99レイテンシ、
  時間帯別エラートレンド、Rate limit到達件数（429）。
- **FR-007**: クエリ結果はリクエストテナントのデータのみを含まなければならない（MUST）。
- **FR-008**: 指定期間に該当データがない場合はゼロ値を返さなければならない（MUST）。

**異常検知（P3）**

- **FR-009**: システムは以下のルールを評価し、条件を満たした場合にアラートを生成しなければならない（MUST）:
  - 5分間の DENY イベント件数が閾値（デフォルト10件）を超過
  - 同一ユーザーが60秒以内に5回連続認証失敗
- **FR-010**: `ListAlerts` gRPC エンドポイントは未解決アラートの一覧を返さなければならない（MUST）。
- **FR-011**: アラートには検知ルール名・発生時刻・関連テナント/ユーザー情報を含めなければならない（MUST）。

**Webhook 通知（P4）**

- **FR-012**: テナントは Webhook URL を1件登録できなければならない（MUST）。
- **FR-013**: 新規アラート生成時、登録済み Webhook URL に POST 通知を送信しなければならない（MUST）。
- **FR-014**: Webhook 送信失敗時は最大3回リトライし、失敗は記録しなければならない（MUST）。

**横断要件**

- **FR-015**: 全 gRPC エンドポイントはリクエスト数・レイテンシ・エラー率をメトリクスとして公開しなければならない（MUST）。
- **FR-016**: 全イベント操作は構造化ログとして記録しなければならない（MUST）。
- **FR-017**: シークレット（DB接続文字列等）はコードに埋め込んではならない（MUST NOT）。

### 主要エンティティ

- **AuthEvent（認証認可イベント）**: システムに投入される最小記録単位。
  `tenant_id`, `user_id`, `service`, `resource`, `action`, `decision`,
  `reason_code`, `latency_ms`, `source_ip`, `trace_id`, `timestamp` を持つ。
- **MetricsSummary（メトリクス集計）**: 期間・テナントを軸に集計したスナップショット。
  ALLOW/DENY数・成功率・レイテンシ分位点を含む。
- **Alert（アラート）**: 異常検知ルールが発火した結果。
  ルール名・重要度・発生時刻・関連ユーザー/テナント・解決フラグを持つ。
- **WebhookConfig（Webhook設定）**: テナントが登録する通知先。
  URL・有効フラグ・最終送信時刻を持つ。
- **Tenant（テナント）**: データ分離の基本単位。
  システム内の全エンティティは `tenant_id` によって分離される。

## 成功基準 *(必須)*

### 測定可能なアウトカム

- **SC-001**: `IngestEvent` のレスポンスタイムは、
  通常負荷（1,000イベント/秒）において99%のリクエストが50ミリ秒以内に完了する。
- **SC-002**: `GetMetrics`（30日分集計）のレスポンスタイムは、
  99%のリクエストが200ミリ秒以内に完了する。
- **SC-003**: 異常検知ルールが条件を満たしてから5分以内にアラートが生成される。
- **SC-004**: Webhook 通知は、アラート生成から30秒以内に送信が開始される。
- **SC-005**: テナント分離の検証テストにおいて、
  テナントAのクエリがテナントBのデータを1件も返さないことを確認できる。
- **SC-006**: システム再起動後も、再起動前に記録されたイベントとアラートが参照できる（永続性）。
- **SC-007**: gRPCクライアント（grpcurl等）から全エンドポイントを呼び出し、
  期待通りのレスポンスが得られることを手動デモで示せる。
- **SC-008**: Envoy の設定変更のみで JWT検証とレート制限を有効/無効に切替可能であることを示せる。

## 前提・除外範囲

### 前提

- Envoy がJWT検証・レート制限・TLS終端を担い、Rustサービス側では実装しない。
- MVP のデータストアは PostgreSQL 1本。TiDB への移行は Port 抽象化で後工程に委ねる。
- 最初は `collector`（イベント受信）と `aggregator`（集計・検知）の2サービス構成で実装する。
- 認証認可イベントは外部アプリケーション（クライアント）がプッシュする形式とする。
- 異常検知の閾値はシステム全体で固定値（環境変数で変更可能）とする。

### 除外範囲（MVP外）

- フロントエンドUI・ダッシュボード画面
- テナント別のレート制限閾値カスタマイズ
- SLAレポート・請求連携
- イベント重複排除（`trace_id` 重複時の冪等処理）
- `detector`・`notifier`・`query-api` の独立サービス化（フェーズ2以降）
- TiDB アダプター実装
- Kubernetes デプロイメント（Docker Compose で完結）
